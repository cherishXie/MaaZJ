## 上游目标

重构后的项目应以 MaaFramework 生态的当前主线为目标，而不是只修补现有模板残留。

截至 2026-06-06 的观测值：

- `MaaXYZ/MaaFramework` 最新 release 为 `v5.10.5`，发布于 2026-05-19。
- `MaaXYZ/MFAAvalonia` 最新 release 为 `v2.12.1`，发布于 2026-05-01。
- `MaaXYZ/MaaPracticeBoilerplate` 是 MaaFramework 通用项目模板，默认分支为 `main`，截至 2026-06-06 近期仍有更新。
- `MaaXYZ/MaaHub` 是 MaaFramework 相关 skill、pipeline、custom、experience 的社区汇总站，可作为 AGENTS 与 skill 知识组织参考。

这些观测值会过期。任何真正升级或实施前，必须重新确认：

- MaaFramework 最新稳定 release、artifact 命名和运行时要求。
- MFAAvalonia 最新稳定 release、artifact 命名、ProjectInterface 支持能力和配置变化。
- MaaPracticeBoilerplate 最新模板结构、CI、install/configure 脚本、README 和资源组织方式。
- MaaHub 中是否已有可复用的 MaaFramework skill、pipeline 范例、custom 代码或开发经验。

## 设计目标

为 MaaZJ 后续重构与升级提供稳定、低噪声的 AI 上下文系统，让后续 agent 能快速理解项目真实运行入口、资源结构、验证方式、高风险边界和上游对齐目标。

方案分为两层：

- `AGENTS.md`：常驻上下文，负责仓库导航、安全规则、优先级规则和上游对齐目标。
- 项目 skill：按需加载的流程知识，负责复杂重构、ProjectInterface 维护、MaaFramework/MFAAvalonia 升级、MaaPracticeBoilerplate 对齐、MaaHub 参考检索和资源分析工具等任务。

参考 MaaHub 的模式：根 AGENTS 保持短小，主要说明 MaaFramework 专属概念、目录入口和检索方向；详细规则放到独立参考文档或 skill references 中按需加载。

## 全局语言规则

后续 AGENTS 与 skill 应加入以下规则：

- 当前项目下新增或修改的说明文档、设计文档、review 结论、任务总结，默认优先使用中文。
- 文件名、路径、API 名称、MaaFramework 字段名、命令、错误信息、代码标识保留原文。
- 当上游规范、库文档或用户明确要求英文时，可以使用英文；否则面向项目维护者的文字输出优先中文。
- 后续 agent 的最终回复和中间说明也应优先中文，除非用户另有要求。

## 当前项目事实

- MaaZJ 是 MaaFramework 资源项目，不是传统应用代码库。
- 主要业务行为位于 `assets/resource/pipeline/` 与 `assets/resource/image/`。
- `assets/interface.json` 是当前运行与发布配置的最高优先级来源。
- 根目录 `interface.json` 与 `assets/interface.json` 不一致，除非任务明确针对根目录文件，否则不要把根目录 `interface.json` 当作运行真相。
- `install.py` 会复制 `assets/interface.json` 到 `install/interface.json`，随后用 `json.load` 读取并写入版本号。这进一步证明 `assets/interface.json` 是发布入口，且它必须保持严格 JSON。
- pipeline 文件中存在 `//` 注释等 JSONC-like 内容，MaaFramework 可接受；不要用严格 JSON 工具批量格式化 pipeline。
- `assets/interface.json` 配置了启动 `{PROJECT_DIR}/agent/main.py`。
- `agent/main.py` 会启动 `AgentServer`，并通过 import 注册 `地图定位检测` custom action 与 `my_reco_222` custom recognition。
- 截至 2026-06-06，未在 pipeline 中找到对 `地图定位检测` 或 `my_reco_222` 的引用；修改前应重新搜索。因此 agent 不是“没用”，而是“已配置启动，但业务接入未证实”。
- `deps/` 与 `install/` 应视为 vendor/generated runtime artifacts，除非用户明确要求更新打包、运行时或发布产物。

## AGENTS 设计

### 1. 根目录 `AGENTS.md`

定位：所有 agent 进入仓库后的第一屏。

建议控制在 100-170 行，写成中文为主，路径与字段名保留原文。

#### 最高优先级规则

root `AGENTS.md` 最前面应写明：

```md
最高优先级：运行与发布配置以 `assets/interface.json` 为准。不要把根目录 `interface.json` 当作 canonical 配置，除非任务明确要求处理它。`install.py` 会复制 `assets/interface.json` 到 `install/interface.json`，并使用 `json.load` 读取，因此 `assets/interface.json` 必须保持严格 JSON。
```

#### 上游对齐规则

root `AGENTS.md` 应加入一小段：

```md
长期目标：重构后的项目应对齐 MaaXYZ/MaaFramework、MaaXYZ/MFAAvalonia 和 MaaXYZ/MaaPracticeBoilerplate 的更新版本，并参考 MaaXYZ/MaaHub 的 skill/pipeline/custom/experience 组织方式。执行升级前必须重新确认上游最新 release、模板结构和文档变化，不要依赖 AGENTS 中的历史版本号。
```

#### 推荐章节

- 项目定位。
- 最高优先级运行真相。
- 文档与回复语言规则。
- 上游对齐目标。
- 目录地图。
- 安全编辑规则。
- 验证命令。
- Python agent 当前状态。
- 重构优先级。
- 升级注意事项。

#### 必须包含的指令

- 项目文档与后续任务输出默认优先中文。
- 运行行为先看 `assets/interface.json`。
- 广泛重构或升级前阅读 `docs/agent_context/PROJECT_UNDERSTANDING_2026-06-06.md`。
- 修改资源后运行 `python .\check_resource.py .\assets\resource\`。
- `assets/interface.json` 按严格 JSON 处理。
- `assets/resource/pipeline/` 下 pipeline 可能是 JSONC-like，不要用严格 JSON 工具批量改写。
- 删除或重命名 pipeline 节点前，搜索 `assets/interface.json` 与所有 pipeline 文件中的引用。
- 不要移除重复账号/服务器 task override，除非已有生成器或受支持的配置机制替代。
- `agent/main.py` 已配置启动；截至 2026-06-06，当前 custom action/reco 的业务接入未证实。修改 Python agent 前重新搜索 pipeline 引用。
- 真正升级 MaaFramework、MFAAvalonia 或 MaaPracticeBoilerplate 对齐前，重新确认上游最新状态。
- 参考 MaaHub 时，只把适合 MaaZJ 的范式引入本项目，不要机械复制社区样例。
- 避免编辑 `deps/` 与 `install/`，除非任务明确涉及打包、运行时或发布产物。

### 2. `assets/AGENTS.md`

定位：指导 `assets/` 内 ProjectInterface、resource bundle、图片、OCR 和 agent 配置关系。

关键规则：

- `assets/interface.json` 是 release-facing 配置，必须是严格 JSON。
- `resource` 路径保持与 `{PROJECT_DIR}/resource` 兼容。
- 不随意重命名图片资源，因为 pipeline 可能通过文件名或逻辑名称引用它们。
- pipeline 可保留 MaaFramework 兼容的 JSONC-like 写法。
- 若对齐新版 MFAAvalonia，需要重点检查 ProjectInterface 字段、任务展示、controller/resource 配置和 agent 配置是否有新能力或兼容性变化。
- 资源改动完成前运行 `python .\check_resource.py .\assets\resource\`。

### 3. `assets/resource/pipeline/AGENTS.md`

定位：专门指导 pipeline 编辑。

关键规则：

- 尽量保留已有节点名，因为 ProjectInterface tasks 与 overrides 可能引用它们。
- `FARM_` 是当前日常流水线的主要命名空间。
- 重命名或删除节点前，搜索 `assets/interface.json` 与所有 pipeline 文件。
- 谨慎处理 `next`，它会改变跨模块流转。
- 不剥离注释，不用严格 JSON 工具批量格式化 pipeline。
- 若需要工具解析 pipeline，应使用 JSONC-capable parser 或 MaaFramework 资源检查结果作为准绳。
- 对齐新版 MaaFramework 前，先查最新版 pipeline 协议、recognition/action 字段和 custom 接入说明，再决定是否迁移旧写法。
- 可参考 MaaHub 中的 pipeline 范式，但必须先确认与 MaaZJ 游戏流程、资源命名和当前 MaaFramework 版本兼容。

### 4. `agent/AGENTS.md`

定位：说明 Python custom action/recognition 的当前角色。

关键规则：

- `assets/interface.json` 已配置启动 `agent/main.py`。
- `agent/main.py` 会注册 `地图定位检测` 与 `my_reco_222`。
- 截至 2026-06-06，未证实 pipeline 使用这些 registered names；修改前先重新搜索引用。
- 能用 pipeline 表达的自动化优先用 pipeline。
- 只有 pipeline 难以可靠表达时，再使用 Python custom action/recognition。
- 对齐新版 MaaFramework 前，重新确认 Python agent API、custom action/custom recognition 注册方式和运行时 agent binary 要求。
- 新增 custom 逻辑时，必须确保 pipeline 中有对应引用，并安排资源验证或手动 debug 路径。

### 延后创建的 AGENTS

以下文件暂不在 v4 首轮实施中创建：

- `.github/AGENTS.md`：用于 CI、release、artifact 命名、上游下载规则。
- `docs/AGENTS.md`：用于文档规范。
- `assets/resource/image/AGENTS.md`：用于图片采集、命名、引用检查。

其中 `.github/AGENTS.md` 在真正升级 MaaFramework/MFAAvalonia 或重写 release workflow 前应优先补充。

## Skill 设计

review 通过后，按“推荐落地顺序”创建项目专属 skill，不应在批准本方案后立刻跳过首轮 AGENTS 实施。

推荐 skill 名称：

`maazj-maafw-refactor`

用途：

- MaaZJ pipeline 重构。
- `assets/interface.json` 维护或生成。
- 减少账号/服务器 task 重复。
- MaaFramework 升级。
- MFAAvalonia 升级与 ProjectInterface 对齐。
- MaaPracticeBoilerplate 模板结构对齐。
- MaaHub 范式检索与参考。
- 资源分析工具建设。
- pipeline 节点引用与图片使用审计。

### Skill 存放位置决策门

创建 skill initial layout 前，必须先显式决定存放位置：

- 若希望 skill 随仓库版本管理、供所有后续维护者复用，选择仓库内路径。
- 若希望 skill 只供当前用户和当前 Codex 环境调用，选择用户全局 Codex skills 目录。
- 决策结果应记录在实施说明或对应设计版本中，再创建文件。

在未完成该决策前，不创建 `maazj-maafw-refactor/` 目录。

### Initial Layout

初期只创建必要文件：

```text
maazj-maafw-refactor/
|-- SKILL.md
`-- references/
    |-- project-map.md
    |-- pipeline-rules.md
    |-- interface-rules.md
    |-- upstream-alignment.md
    `-- upgrade-checklist.md
```

说明：

- 初期不创建 `agents/openai.yaml`，除非实施时明确需要 UI metadata。
- 初期不创建 `scripts/`，避免还未稳定的流程被过早自动化。
- `SKILL.md` 只放触发说明与 references 导航，不写成长篇项目百科。

### `SKILL.md`

保持简洁，只告诉 agent 在不同任务下读哪个 reference：

- 广泛重构、升级、陌生仓库任务：读 `references/project-map.md`。
- 修改 `assets/resource/pipeline/`：读 `references/pipeline-rules.md`。
- 修改或生成 `assets/interface.json`：读 `references/interface-rules.md`。
- 对齐 MaaFramework、MFAAvalonia、MaaPracticeBoilerplate 或参考 MaaHub：读 `references/upstream-alignment.md`。
- 变更 CI release、`deps/`、`install/`、artifact 下载或打包流程：读 `references/upgrade-checklist.md`。

### `references/upstream-alignment.md`

新增 reference，用于沉淀上游对齐规则：

- 记录实施前必须重新确认的上游仓库、release、artifact 命名和关键文档。
- MaaFramework：关注资源加载、pipeline 协议、recognition/action 字段、agent/custom API、运行时 artifact。
- MFAAvalonia：关注 ProjectInterface 支持、任务配置能力、controller/resource/agent 配置、artifact 命名。
- MaaPracticeBoilerplate：关注模板目录结构、CI、install/configure 脚本、README、资源组织方式。
- MaaHub：关注 skill/pipeline/custom/experience 的组织方式和可参考样例。
- 明确禁止机械复制上游模板或 MaaHub 样例；必须先与 MaaZJ 的 canonical `assets/interface.json`、pipeline 和资源验证结果对齐。

### `references/upgrade-checklist.md`

记录升级流程：

- 先冻结 baseline。
- 依赖变更前运行资源验证。
- 实施前重新确认 MaaFramework、MFAAvalonia 最新稳定 release。
- 修改 workflow 上游引用前确认目标 MFAAvalonia upstream，优先使用 `MaaXYZ/MFAAvalonia`，不要继续沿用旧的非目标上游，除非用户明确要求。
- 尽量分开升级 MaaFramework 与 MFAAvalonia。
- MaaPracticeBoilerplate 对齐应作为模板结构对齐，不应覆盖 MaaZJ 已有业务资源。
- 升级后比较资源加载、task 展示、task 启动行为。
- 注意 CI 下载 latest artifacts 与本地 `deps/` 固定旧 artifacts 的行为差异。

### 未来脚本

至少完成一次人工重构后再添加脚本。

候选脚本：

- `generate-interface-tasks.py`：从紧凑配置生成账号/服务器 ProjectInterface tasks。
- `check-pipeline-refs.py`：检查 pipeline 文件与 `assets/interface.json` 的节点引用。

- `check-upstream-releases.py`：只读查询上游 release 和 artifact 命名，生成升级前检查报告。

## 推荐落地顺序

1. review 并批准 v4 方案。
2. 首轮只实施 root `AGENTS.md` 与 `assets/resource/pipeline/AGENTS.md`。
3. root `AGENTS.md` 中提前加入简短 agent 说明和上游对齐目标。
4. review 首轮 AGENTS 是否足够支撑下一步重构。
5. 按需补充 `assets/AGENTS.md` 与 `agent/AGENTS.md`。
6. 决定 `maazj-maafw-refactor` skill 的存放位置，并记录该决策。
7. 创建 `maazj-maafw-refactor` skill 的 initial layout，包含 `upstream-alignment.md`。
8. 冻结当前 baseline，运行 `python .\check_resource.py .\assets\resource\`。
9. 先处理 ProjectInterface 重复 task 与 canonical 文件问题，减少升级比较噪声。
10. 重新确认 MaaFramework、MFAAvalonia、MaaPracticeBoilerplate、MaaHub 当前状态。
11. 分阶段对齐 MaaPracticeBoilerplate 模板结构、MaaFramework runtime、MFAAvalonia release workflow。
12. 每个阶段后验证资源加载、task 展示、task 启动和打包产物。
13. 完成人工重构验证后，再决定是否加入 future scripts。

## 开放问题与 v4 建议

- 设计目录继续使用 `docs/agent_design/`。该名称清晰，且已经承载 v1/v2/v3/v4 版本。
- 项目 skill 的存放位置仍需在落地第 6 步显式决定。建议默认倾向仓库内可追踪路径，除非目标是仅服务当前用户的个人 Codex 环境。
- 根目录 `interface.json` 建议在后续单独决策：同步、标记 stale、迁移或删除都可能影响用户习惯，不在 AGENTS 实施阶段顺手处理。
- 账号/服务器 task 生成建议先于 MaaFramework/MFAAvalonia 大升级进行。原因是它能降低 ProjectInterface 重复度，减少升级时的比较噪声。
- AGENTS 建议中文为主，保留英文 API、字段、路径和命令原文。
- 对齐 MaaPracticeBoilerplate 时，应只吸收模板结构、脚本和 CI 的新做法，不覆盖 MaaZJ 已沉淀的业务资源。
- 参考 MaaHub 时，应优先提炼范式和工作流，不把社区样例直接视为 MaaZJ 的业务实现。
