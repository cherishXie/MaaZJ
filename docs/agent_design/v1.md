# MaaZJ AGENTS and Skill Design v1

Version: v1

Status: draft for review

Scope: design only. Do not implement AGENTS files, project skills, scripts, or refactor work until this design is reviewed and approved.

## Design Goal

Prepare MaaZJ for future refactor and upgrade work by giving AI coding agents a stable, low-noise project context system.

The design uses two layers:

- AGENTS files: always-on repository guidance for safe navigation and edits.
- Project skill: on-demand procedural guidance for complex MaaFramework refactor, ProjectInterface maintenance, and upgrade work.

This follows the pattern observed in MaaXYZ/MaaHub: keep root AGENTS short and navigational, then place detailed MaaFramework-specific knowledge in separate references that agents load only when needed.

## Current Project Facts To Preserve

- MaaZJ is primarily a MaaFramework resource project, not a conventional application codebase.
- Most runtime behavior lives under `assets/resource/pipeline/` and `assets/resource/image/`.
- `assets/interface.json` is the current runtime and release source of truth.
- Root `interface.json` differs from `assets/interface.json` and should not be treated as canonical unless the task explicitly concerns it.
- `install.py` copies `assets/interface.json` into `install/interface.json`.
- Resource changes should be validated with `python .\check_resource.py .\assets\resource\`.
- Some pipeline files may use JSONC-like syntax accepted by MaaFramework. Do not mechanically rewrite them with strict JSON assumptions.
- `agent/` currently contains Python AgentServer template-style custom action/recognition code, but current pipeline usage appears to be almost entirely resource-driven.
- `deps/` and `install/` should be treated as vendor/generated runtime artifacts unless the user explicitly asks to update packaging or bundled runtime files.

## Proposed AGENTS Layout

### 1. Root `AGENTS.md`

Purpose: first screen for every AI agent entering this repository.

Keep it short, preferably 80-150 lines.

Recommended sections:

- Project overview.
- Source-of-truth files.
- Directory map.
- Safe edit rules.
- Validation commands.
- Refactor priorities.
- Upgrade cautions.

Key instructions to include:

- Start from `assets/interface.json`, not root `interface.json`, for runtime behavior.
- Read `docs/agent_context/PROJECT_UNDERSTANDING_2026-06-06.md` when planning broad refactor or upgrade work.
- Validate resource edits with `python .\check_resource.py .\assets\resource\`.
- Preserve MaaFramework-compatible JSONC behavior.
- Do not remove duplicated account/server task overrides until a replacement generation or configuration mechanism exists.
- Avoid editing `deps/` and `install/` unless packaging/runtime update is the explicit task.

### 2. `assets/AGENTS.md`

Purpose: guide changes inside Maa resource and interface assets.

Recommended sections:

- `assets/interface.json` ownership.
- Resource bundle structure.
- Relationship between ProjectInterface, pipeline files, image templates, OCR models, and Python agent.
- Formatting and encoding notes.
- Validation checklist.

Key instructions to include:

- Treat `assets/interface.json` as release-facing configuration.
- Keep `resource` paths compatible with `{PROJECT_DIR}/resource`.
- Do not rename image assets casually, because pipeline nodes may reference them by filename or logical image name.
- After changes, run resource validation before considering the edit complete.

### 3. `assets/resource/pipeline/AGENTS.md`

Purpose: specialized guidance for pipeline edits.

Recommended sections:

- Pipeline module boundaries.
- Node naming conventions.
- Common Maa node fields.
- `pipeline_override` behavior.
- JSON/JSONC handling.
- Debug and validation workflow.

Module boundaries to document:

- Startup and app launch.
- Server selection.
- User/account switching.
- Farm/daily flow.
- Activity flow.
- Shop.
- Guild.
- Notebook.
- Exploration.
- Account creation / early-game setup.

Key instructions to include:

- Preserve existing node names when possible because ProjectInterface tasks and overrides may reference them.
- Treat `FARM_` nodes as the dominant daily-flow namespace.
- Before renaming or deleting a node, search references in `assets/interface.json` and all pipeline files.
- Be careful with `next` overrides because they can alter cross-module task flow.
- Do not strip comments or reformat JSONC-like files with strict JSON tooling.

### 4. `agent/AGENTS.md`

Purpose: clarify the current role of Python custom action/recognition code.

Recommended sections:

- AgentServer startup path.
- Current custom action and recognition registrations.
- Current integration status.
- When to add or modify custom logic.

Key instructions to include:

- Do not assume Python custom code is active; search pipeline references first.
- Prefer pipeline-only changes for declarative automation when sufficient.
- Use Python custom action/recognition only when pipeline nodes cannot express the behavior reliably.
- If adding custom logic, ensure its registered name is referenced by pipeline resources and covered by a resource validation/manual debug path.

### Deferred AGENTS

The following AGENTS files are useful later, but should not be created in v1 unless review identifies an immediate need:

- `.github/AGENTS.md`: for CI, release, artifact naming, and upstream download rules.
- `docs/AGENTS.md`: for documentation conventions.
- `assets/resource/image/AGENTS.md`: for image capture, naming, and reference hygiene.

## Proposed Skill Layout

Create a project-specific skill after AGENTS v1 is approved.

Recommended skill name:

`maazj-maafw-refactor`

Purpose:

Provide on-demand workflows for MaaZJ refactor and upgrade tasks that are too detailed for always-on AGENTS context.

Trigger contexts:

- Refactoring MaaZJ pipeline modules.
- Maintaining or generating `assets/interface.json` tasks.
- Reducing duplicated account/server task configuration.
- Upgrading MaaFramework or MFAAvalonia.
- Building resource analysis tools.
- Auditing pipeline node references or image usage.

### Skill Files

Recommended structure:

```text
maazj-maafw-refactor/
|-- SKILL.md
|-- agents/
|   `-- openai.yaml
|-- references/
|   |-- project-map.md
|   |-- pipeline-rules.md
|   |-- interface-rules.md
|   `-- upgrade-checklist.md
`-- scripts/
    |-- generate-interface-tasks.py
    |-- check-pipeline-refs.py
    `-- check-image-usage.py
```

Only `SKILL.md` and `references/` should be created initially. Add scripts later when the related workflow is stable enough to automate.

### `SKILL.md`

Keep this file concise. It should tell agents which reference to read for each task type.

Suggested navigation:

- Read `references/project-map.md` before broad refactor, upgrade, or unfamiliar repository work.
- Read `references/pipeline-rules.md` before editing `assets/resource/pipeline/`.
- Read `references/interface-rules.md` before editing or generating `assets/interface.json`.
- Read `references/upgrade-checklist.md` before changing MaaFramework, MFAAvalonia, CI release flow, `deps/`, or `install/`.

### `references/project-map.md`

Condense the current project understanding snapshot into reusable context:

- Runtime path: MFAAvalonia -> `interface.json` -> resource bundle -> MaaFramework pipeline -> optional Python agent.
- Canonical files.
- Main directories.
- Known risks.
- Current refactor targets.

### `references/pipeline-rules.md`

Document pipeline-specific working rules:

- Search-before-rename discipline.
- Node namespace conventions.
- JSONC handling.
- Module boundaries.
- How to reason about `next`, `expected`, recognition, and action changes.
- Validation process.

### `references/interface-rules.md`

Document ProjectInterface maintenance:

- `assets/interface.json` is canonical.
- Root `interface.json` is divergent.
- Account/server task duplication is a known refactor target.
- Preserve `pipeline_override` behavior until a generator or supported MFAAvalonia option replaces it.
- Add or remove tasks only with an explicit runtime intent.

### `references/upgrade-checklist.md`

Document upgrade workflow:

- Freeze baseline first.
- Run resource validation before dependency changes.
- Verify intended MFAAvalonia upstream before changing workflow references.
- Upgrade MaaFramework and MFAAvalonia separately when possible.
- Compare resource loading and task launch behavior after upgrade.
- Treat CI latest-artifact downloads as a divergence risk from local `deps/`.

### Future Scripts

Add scripts only after review and at least one manual refactor pass.

Potential scripts:

- `generate-interface-tasks.py`: generate account/server ProjectInterface tasks from a compact data file.
- `check-pipeline-refs.py`: check node references across pipeline files and `assets/interface.json`.
- `check-image-usage.py`: identify image assets not referenced by pipeline definitions.

## Recommended Rollout Order

1. Review and approve this v1 design.
2. Implement root `AGENTS.md` and `assets/resource/pipeline/AGENTS.md` only.
3. Review whether these two files are enough for the next planned refactor.
4. Add `assets/AGENTS.md` and `agent/AGENTS.md` if needed.
5. Create the `maazj-maafw-refactor` skill with `SKILL.md` and references only.
6. Add scripts after repeated manual workflows prove what should be automated.
